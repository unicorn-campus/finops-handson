## 1. Virtual Machines(VM) — "켜 둔 시간(분)"으로 과금

- **과금 단위**: 컴퓨팅은 **VM이 실행된 전체 분(分) 단위**로 과금됨.  
  초 단위 사용량을 기록하되 청구는 분 단위로 집계함(예: 6분 45초 실행 → 6분 과금).
- **상태별 과금**: 실행(Running) 중에만 컴퓨팅 과금됨.  
  **중지·할당 취소(Stopped/Deallocated)** 상태에서는 컴퓨팅은 과금되지 않으나  
  **연결된 디스크·고정 공용 IP 등 부속 리소스는 계속 과금됨**(→ 2절).
- **가격 결정 요인**: VM 크기(SKU·vCPU·메모리), OS·소프트웨어 라이선스, 리전, 요금 옵션(종량제 / 예약 / Savings Plan / Spot).
- 💡 **최근 변경(2025-06, MS Q&A 안내)**: 짧은 실행 VM에 **최소 5분(300초) 과금**이 적용됨 —  
  5분 미만에 시작·중지해도 5분만큼 과금됨(출처: [MS Q&A - 단시간 VM 과금 안내](https://learn.microsoft.com/en-us/answers/questions/2259765/clarification-on-azure-vm-billing-for-short-runtim)).

> 🔑 **FinOps 포인트**: VM은 "켜 둔 시간"이 비용임 → **미사용 VM 중지/스케줄링**이 즉효.  
> 단, 중지만으로 **디스크 비용은 사라지지 않음** — 디스크 삭제·계층 조정까지 함께 봐야 함.

## 2. 연결 디스크(Managed Disks) — "쓴 만큼"이 아니라 "요청한 만큼" 과금

Azure Managed Disk의 가장 큰 오해가 여기 있음 —  
**과금 기준은 실제 사용량이 아니라 "프로비저닝(요청)한 용량"임.**

- **Premium SSD / Standard SSD / Standard HDD**: **성능 계층(P/E/S 티어)** 으로 과금됨.  
  요청한 용량을 **가장 가까운 상위 티어로 올림(round-up)** 하여 **해당 티어 전액을 과금**함.  
  예: 1TiB Standard SSD → E30 티어로 과금(실제 120GiB만 써도 E30 전액 과금).
- **Ultra Disk / Premium SSD v2**: **1GiB 단위(per-GiB)** 로 과금되며,  
  추가로 **프로비저닝한 IOPS·처리량(Throughput)** 도 과금됨(여전히 요청량 기준, 실사용 무관).
- **스냅샷(Snapshot)**: 예외적으로 **사용된(used) 데이터 크기 기준**으로 과금됨  
  (예: 500GiB 디스크 중 50GiB 사용 → 스냅샷은 50GiB만 과금). 증분 스냅샷이 더 경제적임.
- **빈 디스크(Empty disk)**: 생성 직후 첫 사용(VM 연결·VHD 업로드) 전까지는 메타데이터만 저장되어  
  스토리지 과금이 발생하지 않음. 첫 사용 시점부터 과금 시작됨.

> 🔑 **FinOps 포인트**: "디스크는 요청 용량으로 과금" → **과대 프로비저닝(over-provisioning)이 곧 낭비**.  
> Right-sizing(적정 용량)·미연결(orphaned) 디스크 정리·계층(성능 tier) 하향이 절감 레버임.  
> ※ 티어 산정·올림 규칙은 [Understand Azure Disk Storage billing](https://learn.microsoft.com/en-us/azure/virtual-machines/disks-understand-billing) 참조.

## 3. Azure Database for PostgreSQL (Flexible Server) — 3개 축으로 각각 과금

비용 비중이 큰 관리형 DB로, **컴퓨팅·스토리지·백업이 청구서에 별도 라인 항목**으로 표시됨.

| 과금 항목 | 단위 | 설명 |
|---|---|---|
| **컴퓨팅(Compute)** | **vCore-시간** | 티어(Burstable / General Purpose / Memory Optimized)·vCore 수로 결정, 시간당 과금 |
| **스토리지(Storage)** | **GiB-월** | 프로비저닝한 저장 용량(데이터·임시·트랜잭션 로그 등) |
| **백업 스토리지(Backup)** | **GiB-월** | 프로비저닝 스토리지의 **100%까지 무료**, 초과분만 GiB/월 과금 |

- **고가용성(HA)**: 영역 중복 HA는 주(primary)·보조(secondary) 복제본 **양쪽 모두** 컴퓨팅·스토리지 과금됨(약 2배).
- **절감 옵션**: 예약 용량(Reserved Capacity)으로 종량제 대비 상당 폭 절감 가능(약정 기간에 따라 절감률 상이).

> 🔑 **FinOps 포인트**: 컴퓨팅 티어 right-sizing + 예약, 백업 보존 기간 관리, 불필요 HA 재검토가 핵심.  
> "쓰지 않는 시간대에도 프로비저닝된 컴퓨팅·스토리지는 과금됨"을 인지해야 함.

## 4. Storage Accounts (Blob) — 용량 외에 "작업 수"까지 과금

스토리지 계정은 **여러 차원이 동시에 과금**되는 대표 제품임 —  
용량만 보면 실제 청구액을 예측하기 어려움.

| 과금 차원 | 설명 |
|---|---|
| **저장 용량(Capacity)** | 저장 데이터량(GB/월). **액세스 계층(Hot/Cool/Cold/Archive)** 이 낮을수록 저장 단가↓ |
| **트랜잭션(Operations)** | 모든 읽기/쓰기 등 **작업 횟수**(보통 1만 건 단위 표기). 계층이 낮을수록 작업 단가↑ |
| **데이터 검색(Data Retrieval)** | Cool·Cold·Archive 계층에서 **읽을 때 GB당 검색 요금** 부과 |
| **데이터 전송(Egress)** | 리전 밖으로 나가는 아웃바운드 전송량 |
| **중복 옵션(Redundancy)** | LRS/ZRS/GRS 등 복제 수준에 따라 단가 상이 |

- **계층 트레이드오프**: 계층이 차가울수록(Cool→Archive) **저장 단가는 내려가나 접근(트랜잭션·검색) 비용은 올라감**.  
  Archive는 읽기 전 복원(rehydrate)이 필요하고 시간·비용이 큼.
- 단위 표기: 청구서 GB 라인은 이진 GB(GiB) 기준으로 계산됨.

> 🔑 **FinOps 포인트**: "저장 용량만 최적화"는 함정 —  
> 접근 빈도에 맞춘 **수명 주기(Lifecycle) 계층 이동** + 트랜잭션·검색 패턴까지 함께 봐야 실제 절감됨.

## 5. Azure Monitor (Logs / Log Analytics) — "수집량이 곧 비용"

관측성(Observability) 비용의 핵심은 **Log Analytics 작업 영역으로 보낸 데이터량(수집)** 임.

| 과금 항목 | 단위 | 설명 |
|---|---|---|
| **데이터 수집(Ingestion)** | **GB** | 작업 영역으로 전송된 로그량. 요금제(플랜)별 단가 상이 |
| **데이터 보존(Retention)** | **GB-월** | **31일까지 무료**, 이후 보존 연장분에 대해 GB-월 과금 |
| (부가) 알림·메트릭 등 | 규칙·항목 수 | 알림 규칙, 웹 테스트 등 부가 기능 별도 과금 |

- **로그 플랜(3종)**: **Analytics Logs**(분석용, 단가 최고) · **Basic Logs**(저비용) · **Auxiliary Logs**(최저가) —  
  용도(대화형 분석 필요 여부)에 따라 플랜을 선택해 수집 단가를 조절함.
- **약정 계층(Commitment Tier)**: Analytics Logs 수집에 대해 일정량 약정 시 종량제 대비 절감 가능  
  (Basic·Auxiliary는 정액 GB 단가로 약정 할인 대상 아님).

> 🔑 **FinOps 포인트**: "일단 다 수집" 대신 **필요한 로그만·적정 플랜으로 수집**, 보존 기간 최소화가 절감 레버임.  
> 수집량은 리소스 증가에 비례해 조용히 커지므로 **정기 모니터링**이 필요함.

### 6. AKS (Azure Kubernetes Service) — "관리는 (거의) 무료, 비용은 노드"

- **관리(컨트롤 플레인) 티어**: **Free**(클러스터 관리 무료, 재정 보증 uptime SLA 없음, 개발·소규모) /  
  **Standard**(가동시간 SLA 기본 포함, **클러스터당 시간당 요금**, 프로덕션·최대 5,000노드) /  
  **Premium**(Standard + **24개월 LTS** 장기 지원, 규제 환경).
- **실제 비용의 대부분**: 소비 리소스 — 에이전트 **노드 VM·디스크·부하 분산·네트워크**(표준 Azure 요금으로 별도 과금).
- **함정**: "관리형 K8s라 저렴"이 아니라 **노드 풀 크기·유휴 노드**가 비용임. 청구 대상은 클러스터가 아니라 노드임.

> 🔑 **FinOps 포인트**: 노드 right-sizing · Cluster Autoscaler · **Spot 노드 풀** · 유휴 노드 축소가 절감 레버임.

### 7. 데이터 전송(Egress / 대역폭) — "받는 건 공짜, 내보내는 게 비용"

- **인바운드(수신) 무료**, **아웃바운드(송신) GB 단위 과금**.  
  인터넷 egress는 **월 100GB 무료** 후 구간별 과금(대량일수록 단가↓), **리전(Zone)별 단가 상이**.
- **리전 간(cross-region)**: 아웃바운드 과금·인바운드 무료. **가용성 영역(AZ) 간** 전송도 소액 과금됨.
- **함정**: 아키텍처 도면에 잘 안 보이는 **숨은 비용** — 통신사·미디어처럼 트래픽 대량이면 급증함.

> 🔑 **FinOps 포인트**: 리전 배치 최적화(트래픽 국지화) · **CDN·캐싱** · 불필요 리전 간 복제 축소 ·  
> 사설 연결(Private Link/피어링) 활용으로 egress 경로·비용을 관리함.

---

### 8. ExpressRoute — 온프렘 전용회선, "요금제 선택"이 핵심

- 온프렘↔Azure **전용 회선**. 데이터 요금제 2종 중 선택:
  - **Metered(종량제)**: 포트 월정액 + **아웃바운드 데이터 GB 과금**(사용량 적을 때 유리)
  - **Unlimited(무제한)**: 단일 **고정 월 포트 요금**(인·아웃 데이터 무료, 사용량 클 때 유리)
- **추가 과금**: **가상 네트워크 게이트웨이(ExpressRoute Gateway)** SKU별 시간당 요금, Global Reach·Premium 애드온.
- **함정**: 회선 요금만이 아니라 **게이트웨이 SKU·요금제(Metered/Unlimited) 선택**이 총비용을 좌우함.

> 🔑 **FinOps 포인트**: 실제 트래픽량 기준 Metered↔Unlimited **정기 재평가**, 게이트웨이 SKU 적정화가 핵심임.

---

### 9. 게이트웨이·방화벽 (App Gateway · NAT Gateway · Azure Firewall) — "고정 + 처리량" 이중

공통 패턴: **고정 요금(시간당 배포/리소스) + 사용량 요금(처리 데이터 GB)** 의 **이중 과금**임.

| 서비스 | 고정 요금 | 사용량 요금 |
|---|---|---|
| **Application Gateway v2** | 시간당 기본 | **용량 단위(CU)** 소비(시간당·처리량 비례) |
| **NAT Gateway** | 리소스 시간당 | **처리 데이터 GB당** |
| **Azure Firewall** | 배포 시간당 | **검사 데이터 GB당** |

- **함정**: 트래픽이 늘수록 **처리 GB 요금**이 조용히 증가함(특히 NAT Gateway·Firewall의 데이터 처리 요금).

> 🔑 **FinOps 포인트**: 불필요 인스턴스 통합 · 트래픽 경로 최적화(불필요한 방화벽 통과 회피) · 미사용 게이트웨이 제거.

---

### 10. Azure SQL Database — 구매 모델 + 서버리스 자동 일시중지

- **구매 모델**: **DTU**(컴퓨팅·스토리지·IO 번들, 단순) vs **vCore**(컴퓨팅·스토리지 분리, 유연 + AHB·예약 적용).
- **컴퓨팅 계층(vCore)**: **프로비저닝(시간당 고정)** vs **서버리스**.
  - 서버리스: **초당 vCore 사용량 과금**(min/max vCore 범위 설정),  
    **비활성 시 자동 일시중지 → 컴퓨팅 0, 스토리지만 과금**(활동 시 자동 재개, 기본 지연 60분).
- **함정**: 상시 저부하 DB에 프로비저닝 고정 티어면 낭비 / 서버리스는 자주 깨우면 오히려 비쌀 수 있음.

> 🔑 **FinOps 포인트**: 사용 패턴에 맞춰 서버리스↔프로비저닝 선택, vCore+예약/AHB(단, **서버리스엔 예약·AHB 미적용**),  
> 다중 DB는 **탄력적 풀(Elastic Pool)** 로 통합해 단가를 낮춤.

---

### 11. Cosmos DB — "처리량(RU/s)"이 비용의 중심

- 과금 = **처리량(RU/s)** + **저장(GB/월)** (+ 백업). RU(Request Unit)는 읽기·쓰기·쿼리 비용의 정규화 단위임.
- **처리량 모드 3종**:
  - **프로비저닝(표준)**: 수동으로 RU/s 확보(시간당) — 예측 가능 부하
  - **Autoscale**: 설정 최대치 내 자동 조정 — 변동 부하
  - **Serverless**: 실제 소비 RU만 과금 — 간헐·소규모
- 예약 용량(Reserved Capacity)으로 최대 63% 절감 가능.
- **함정**: **RU/s 과대 프로비저닝**이 대표 낭비임. 비효율 인덱싱·부적절 파티션 키도 RU 소비를 키움.

> 🔑 **FinOps 포인트**: 부하 패턴별 모드 선택(Autoscale/Serverless) · RU/s right-sizing · 인덱싱 정책·파티션 키 최적화 · 예약 적용.

---

### 12. Azure OpenAI Service — "토큰 vs PTU" (KT–MS 파트너십 핵심)

- **종량제(Standard/PAYG)**: **토큰 단위** 과금 — **입력·출력 토큰을 각각** per 1K/1M tokens로 과금(모델별 단가 상이,  
  일반적으로 **출력 토큰 단가가 더 높음**).
- **PTU(Provisioned Throughput Unit)**: 배포한 PTU 수 × **시간당($/PTU/hr)** — 처리량·지연 보장,  
  **Azure 예약(월/연 약정)** 으로 대폭 할인. 지속·대량 워크로드에 유리.
- 기타: **Batch**(비동기 대량, 할인) · 파인튜닝(학습 + 호스팅 시간당).
- **함정**: 프롬프트·출력이 길수록 토큰 급증 / PTU는 저활용 시 낭비(예약분을 못 채움).

> 🔑 **FinOps 포인트**: 지속 대량이면 **PTU+예약**, 간헐이면 **종량제** · 프롬프트·컨텍스트 최적화 ·  
> 작은 모델 우선 선택 · 캐싱·Batch 활용으로 토큰·처리량 비용을 낮춤.

---

### 13. 빅데이터 (Azure Databricks · Event Hubs)

**Azure Databricks — DBU + 하부 인프라 "이중 과금"**
- 과금 = **DBU(Databricks Unit)** + **하부 Azure 리소스(VM·스토리지·네트워크)**.
- DBU: 워크로드 유형·티어별 **DBU-시간(초당 과금)** — 예: All-purpose > Jobs > Jobs Light, Serverless SQL 등 단가 상이.
- **핵심**: DBU는 Databricks 소프트웨어 요금이고, **VM 등 인프라는 Azure 구독으로 별도 청구**(인프라가 DBU의 50 ~ 100% 수준).
- **함정**: DBU만 보고 하부 VM 비용을 간과하면 실제 청구가 예상의 2배가 됨.

**Azure Event Hubs — 처리 단위 선구매**
- 티어별 처리 단위: Standard=**처리량 단위(TU)** · Premium=**처리 단위(PU)** · Dedicated=**용량 단위(CU)**.
- 1 TU: 수신 1MB/s(또는 1,000 events/s) · 송신 2MB/s(또는 4,096 events/s). **TU 시간당 과금 + 보존 저장 초과분·Capture 별도**.
- **함정**: TU 과대 설정, Capture·장기 보존의 추가 비용.

> 🔑 **FinOps 포인트**: Databricks는 Jobs/Serverless 적정 선택 · Spot · 오토터미네이션 · 클러스터 정책 /  
> Event Hubs는 TU·PU right-sizing · 오토인플레이트(Auto-inflate) 상한 관리로 처리량 과금을 통제함.

## ✅ 요약 — FinOps 관점 3줄

- **VM**: "켜 둔 분(分)"이 비용 → **끄기·스케줄링**. 단, 중지해도 **디스크·IP는 계속 과금**됨.
- **디스크**: "쓴 만큼"이 아니라 **"요청한 용량"** 이 비용 → **과대 프로비저닝·미연결 디스크가 낭비**.
- **DB·Storage·Monitor**: **다차원 과금**(컴퓨팅+스토리지+백업 / 용량+작업+검색 / 수집+보존) →  
  단일 지표가 아니라 **과금 포인트별로 나눠 봐야** 실제 비용을 읽고 줄일 수 있음.

**확장 제품(대기업·통신사) 관점**
- **AKS·네트워킹**: AKS는 "클러스터가 아니라 **노드**가 비용", 데이터 전송은 "**아웃바운드**가 비용",  
  ExpressRoute·게이트웨이·방화벽은 "**고정(시간당) + 처리량(GB)** 이중 과금" → 트래픽이 조용히 비용을 키움.
- **데이터·AI**: Azure SQL·Cosmos DB는 "**적정 처리량·서버리스/Autoscale 선택**", Azure OpenAI는 "**토큰 vs PTU**",  
  Databricks는 "**DBU + 하부 VM 이중**" → 소비 단위(RU·토큰·DBU)를 이해하고 right-sizing 해야 함.

> ⚠️ 본 문서의 구체 금액·비율은 시점·리전·계약별로 변동되므로,  
> 반드시 각 절의 **1차 출처(요금 페이지·Pricing Calculator)** 로 최신 값을 확인 후 의사결정에 활용함.
