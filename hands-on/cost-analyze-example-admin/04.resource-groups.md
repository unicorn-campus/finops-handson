# 04. Resource groups — 리소스 그룹(앱·워크로드)별 비용(어느 애플리케이션이 썼는가)

> 페이지: Resource groups · 데이터 범위: 청구기간 2024-06-01 ~ 2024-07-08 · 필터 전체(All) · 통화 USD(샘플)  
> 원본: CostManagementConnector.pbix (FinOps Toolkit) · Inform 단계 비용 가시화  
> 📌 한 줄 요약(TL;DR): ahbtest RG가 최대이고 RG→리소스 드릴다운으로 원인 추적 가능하나, RG 없는 비용(구매·용량)은 누락됨.

![](./04.resource-groups.png)

## 1. 개요
- 구독보다 한 단계 세밀하게 "어느 리소스 그룹(=보통 애플리케이션·워크로드 단위)이 비용을 쓰는가"를 보는 화면임  
- 데이터 범위: 청구기간 `2024-06-01 ~ 2024-07-08` / 필터 모두 All / 통화 USD(샘플)

## 2. 화면 구조·차트 읽는 법
- 상단: 동일 필터 + Effective cost **81.1천** / Total savings **1.08천**  
- 가운데: **일자별 누적 막대** — 리소스 그룹별 색깔. 큰 보라색 띠 = **ahbtest** RG가 지배적  
- 하단 표: **리소스 그룹별**(괄호 안에 소속 구독 표기), **개별 리소스까지 드릴다운** 가능

### 표에서 눈여겨볼 점 — 계층 드릴다운
- 각 RG 앞의 `⊞` 아이콘 → 펼치면 그 안의 개별 리소스가 나옴  
- 예: `ahbtest (Cost Management Research)` 펼침 →  
  - `ahbfoo` (SQL Managed Instance)  
  - `AHBTEST02` (Data Factory)  
  - `ahbtest1` (SQL servers)  
- **RG → 리소스**까지 파고들어 비용 원인을 리소스 단위로 특정 가능

### 중요한 관찰 — 차트 눈금이 낮음 (~600/일)
- 앞 페이지(Services·Subscriptions)는 하루 ~2천이었으나, 이 RG 차트는 하루 ~600 수준  
- 이유: **모든 비용이 리소스 그룹에 매핑되지 않기 때문**  
  - 예약(RI)·Marketplace 구매, 구독 레벨 요금, 용량(capacity) 기반 과금 등은 RG가 없음  
  - 그래서 범례에 **`(공백)` = 리소스 그룹 없음** 항목이 존재  
- 즉 RG 뷰만으로는 총비용이 다 안 보임 → 총액은 상단 카드(81.1천)로 확인

## 3. 분석 요약
> What · 데이터가 보여준 사실(해석 배제)

- 일자별 누적 막대에서 **큰 보라색 띠 = ahbtest RG**가 지배적으로 RG 레벨 최대 비용원  
- 차트 눈금이 **하루 ~600 수준**으로, 앞 페이지(하루 ~2천)보다 낮음  
- 낮은 이유: 예약(RI)·Marketplace 구매·구독 레벨 요금·용량(capacity) 과금 등은 RG가 없어 매핑되지 않음  
- 범례에 **`(공백)` = 리소스 그룹 없음** 항목이 존재  
- `ahbtest (Cost Management Research)` 드릴다운 시 `ahbfoo`(SQL Managed Instance)·`AHBTEST02`(Data Factory)·`ahbtest1`(SQL servers) 등 개별 리소스 확인 가능

## 4. 시사점
> So what · 사실의 의미·비용 리스크

- **ahbtest RG(보라)가 RG 레벨 최대** → 애플리케이션 단위 최적화 1순위  
- **RG=애플리케이션 경계** — 각 RG가 앱/워크로드면 이 표가 곧 앱별 비용  
- **리소스 드릴다운 가능** — 급증 원인을 개별 리소스(특정 SQL·Data Factory 등)까지 추적 가능  
- **`(공백)` 주의** — RG 없는 비용(구매·용량 등)은 RG 뷰에서 누락 → 배분 시 별도 처리 필요한 사각지대

## 5. 권고사항
> Now what · Inform 단계 실행 행동(실행은 Optimize 이관 명시)

- **ahbtest RG 우선 최적화** — RG를 펼쳐 SQL Managed Instance·Data Factory 등 개별 리소스별 비용 규명 → Optimize 이관  
- **RG 없는 비용 별도 배분 규칙 수립** — `(공백)` 항목(구매·용량성 비용)은 구독·용도별로 별도 배분 처리  
- **RG 태깅·명명 규칙 정비** — RG가 앱/워크로드 경계와 일치하도록 정리해 앱별 비용 가시성 확보  
- **배분·책임 계층 운영** — 원인 추적은 리소스, 배분·책임은 구독/RG 단위로 운영(아래 계층 참조)

## 6. 용어·출처

### 용어
- **Resource group(리소스 그룹, RG)**: 보통 애플리케이션·워크로드 단위로 리소스를 묶는 논리 그룹  
- **드릴다운(drill-down)**: 상위 항목(RG)을 펼쳐 하위 항목(개별 리소스)까지 파고드는 분석  
- **`(공백)` = 리소스 그룹 없음**: RI·Marketplace 구매·구독 레벨·용량 과금 등 RG에 매핑되지 않는 비용

### 보충 — 배분 계층 정리
```
구독(팀) → 리소스 그룹(앱/워크로드) → 리소스(개별 자원)
 03번        04번(이 페이지)            04번 드릴다운
```
아래로 갈수록 세밀 → 원인 추적은 리소스, 배분·책임은 구독/RG 단위가 실용적임.
