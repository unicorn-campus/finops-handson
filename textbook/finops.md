# Part 1. FinOps 개요

## 1. 정의와 필요성

- **FinOps(Financial Operations)**: Finance + DevOps의 합성어로, 기술 비용 최적화와 투자 가치 극대화를 위해
  재무·비즈니스·IT 조직이 데이터 기반으로 협업하는 체계
- **왜 필요한가**: 사용량 기반 비용 모델로 전환되며 비용 예측·최적화가 어려워지고,
  비용 책임이 조직 전체로 분산되어 투자 가치 증명이 어려워졌기 때문

| 구분 | 전통적 IT (CapEx) | 클라우드 (OpEx) |
|------|-------------------|-----------------|
| 비용 모델 | 고정비 — 장비 구매 후 감가상각 | 변동비 — 사용량에 따라 과금 |
| 비용 예측 | 예측 가능(사전 계획 기반) | 예측 어려움(수요 변동) |
| 비용 책임 | IT 부서 집중 | 분산(엔지니어·PO·재무) |
| 최적화 시점 | 구매 시점 1회 결정 | 지속적 모니터링·최적화 필요 |

- **핵심 수치**: 클라우드 지출 중 낭비 약 1/3(Flexera State of the Cloud Report),
  FinOps 도입 시 월간 비용 20 ~ 30% 절감(FinOps Foundation)
- FinOps Foundation은 2026년 초 미션을 "Cloud"에서 "Technology"로 확장하여 SaaS·AI 라이선스까지 범위 확대
  (출처: finops.org/framework)

## 2. 골든써클(WHY-HOW-WHAT)

- **WHY(왜)**
  - 투자 가치 극대화: 기술 투자가 비즈니스 성과로 이어지는지 증명
  - 기술 비용 최적화: 클라우드·SaaS·AI·라이선스 등 기술 지출에서 낭비 제거
- **HOW(어떻게)** — 3단계 전략
  - 보이기(Inform): 비용 데이터를 수집·시각화하여 누가·어디에·얼마를 쓰는지 가시화
  - 줄이기(Optimize): 가시성을 기반으로 낭비 제거 및 비용 효율성 개선
  - 체계화(Operate): 지속 가능한 운영 체계로 정착되도록 구조화
- **WHAT(무엇을)** — 실행 역량
  - 비용 관리 체계: 수집 → 분석 → 탐지 → 대응 → 예측 체계 구축
  - 태그 기반 비용 추적/할당: 조직·환경 등으로 태깅하여 비용 추적·할당
  - Right-sizing: 과잉 리소스를 적정 사양으로 조정, 미사용 리소스 제거
  - 스케일링 정책 최적화: HPA/VPA/노드 오토스케일링 정책 튜닝
  - 약정 할인 의사결정: RI/SP/SI 등 워크로드 특성에 맞는 할인 전략 적용
  - FinOps 자동화: 탐지 → 권고 → 승인 → 적용 → 검증 파이프라인 구축
  - KPI 관리: 비즈니스 성과의 단위당 비용 측정 포함
  - 협업 문화와 거버넌스 체계: 정기적 비용 리뷰·비용 관리 프로세스 수립

> 약어: RI(Reserved Instance), SP(Savings Plans), SI(Spot Instance),
> HPA(Horizontal Pod Autoscaling, Pod 수 자동 증감), VPA(Vertical Pod Autoscaling, CPU/Mem 자동 증감)

## 3. FinOps 6대 원칙 (COVERS)

| 약자 | 원칙 | 핵심 내용 |
|------|------|-----------|
| **C** | Collaboration(팀 간 협업) | 엔지니어링·재무·비즈니스팀이 함께 의사결정, 사일로를 허물고 공통 목표 협력 |
| **O** | Ownership(오너십) | 엔지니어·PO가 클라우드 지출에 책임 보유("내 서비스 비용은 내가 관리") |
| **V** | Value-driven(가치 중심) | 단순 절감이 아닌 투자 가치 극대화가 목표, 성과 기여도 측정 |
| **E** | Elastic Utilization(탄력적 활용) | 미사용 용량 최소화, 수요에 따라 탄력적으로 운영 |
| **R** | Right-time Report(적시 리포팅) | 투명한 비용 데이터를 실시간에 가깝게 제공, 접근성·적시성·정확성 보장 |
| **S** | Steering(중앙 운영 주도) | 중앙 FinOps 팀(CCoE)이 거버넌스 드라이브, 정책은 중앙·실행은 각 팀(연합 거버넌스) |

## 4. FinOps 성숙도 모델

- **Crawl(기초) — 비용 가시성 확보**
  - 태깅 전략 수립·적용, 청구 대시보드·월간 레포팅
  - 이해관계자 식별·역할 정의, FinOps 전담 조직(CCoE) 구성, 기본 KPI 정의
- **Walk(발전) — 최적화 실행**
  - 자동화된 비용 분석 도입, Right-sizing, RI/Savings Plans
  - Showback(팀별 쓴 비용 '보여주기')
- **Run(고도화) — 자율 운영**
  - 단위 경제(매출·고객 1건당 비용) 도입, 실시간 비용 거버넌스, 예측 기반 의사결정
  - Chargeback(쓴 비용을 팀에 실제 '청구·부과'), FOCUS 멀티클라우드 통합, FinOps 문화 내재화

> **2025 프레임워크 업데이트**: 성숙도를 전사 단일 점수가 아닌 Ownership별·Capability별로 측정하도록 변경 →
> 팀별 3 ~ 6개월 주기 재평가 권장

- **한국 시장 특징**: 대부분 Crawl ~ Walk 수준이며 Run 도달 기업은 소수.
  국내 CSP + 글로벌 CSP 조합 구조로 멀티클라우드 비용 통합 관리 복잡도가 높음
- **국내 도입 사례 — 우아한형제들(배달의민족)**: 단순 절감이 아닌 비즈니스 가치 극대화 전략으로 실천,
  단위 경제(트래픽 1건 처리 비용) 기반 성과 측정, 기술 블로그로 경험 공유

## 5. FinOps 3가지 전략(반복적 순환 구조)

- 보이기 → 줄이기 → 체계화의 세 단계는 선형 완료가 아닌 **반복적 순환(Iterative Cycle)** 구조
- 성숙도가 높아질수록 더 깊은 수준에서 각 단계를 재실행

| 단계 | 목표 | 세부 활동 |
|------|------|-----------|
| 보이기(Inform) | 비용 가시성 확보 | 비용 데이터 수집(CSP 청구 중앙집중화), 대시보드, 비용 할당(태그), 이상 비용 탐지 |
| 줄이기(Optimize) | 비용 효율화 실행 | 미사용 리소스 제거, Right-sizing, 약정 할인 적용, 스케일링 정책 최적화 |
| 체계화(Operate) | 지속적 거버넌스 운영 | 단위 경제·KPI 관리, 자동화, 협업 문화·거버넌스 |

> "측정할 수 없으면 관리할 수 없다"(피터 드러커) — 보이기 단계가 가장 기본이며,
> 비용 가시성 없이는 최적화도 거버넌스도 불가능함

## 6. [참고] 주요 제품별 요금 체계

- 과금은 '쓴 만큼(사용량)'과 '확보한 만큼(고정)'으로 나뉨 — 둘을 구분해야 비용이 보임
  - 사용량 = 실행·전송·작업·소비에 따라 변동 / 고정 = 확보·프로비저닝·예약(안 써도 과금)

| 제품 | 사용량 과금 | 고정 과금 | 핵심 포인트 |
|------|-------------|-----------|-------------|
| Virtual Machines(VM) | VM 크기 × 할당 시간 | 없음 | 켠 시간이 비용, Deallocated 해도 디스크·IP 과금 |
| Managed Disks | 없음¹ | 프로비저닝 용량(GB) + IOPS·처리량 | 실사용 무관 — 100GB 할당·10GB 사용도 100GB 과금 |
| Azure DB for PostgreSQL | 컴퓨팅 vCore-시간·초과 백업 | 프로비저닝 스토리지 GiB-월 | 컴퓨팅·스토리지·백업 별도, HA 구성 시 대기 서버 비용 추가 |
| Storage(Blob) | 평균 용량·작업 수·검색·egress | 없음 | 실사용 용량 과금, 저장 계층별 작업·검색·조기 삭제 비용 |
| Azure Monitor(Logs) | 로그 수집·보존 GB-월·검색·복원 | 일일 수집 약정 계층(선택) | 약정 초과분 별도 청구, 플랜·보존기간에 따라 차이 |
| AKS | 노드 VM 실행시간·디스크·네트워크 | Standard/Premium 티어 시간당 관리비 | Free 티어만 관리 노드 무료, 실제 비용은 워커 노드 |
| 데이터 전송(Egress) | 인터넷·리전 간 아웃바운드 GB | 없음 | 인바운드 무료·아웃바운드 과금(인터넷 송신 월 100GB 무료) |

> ¹ Standard SSD/HDD는 소액 I/O(트랜잭션) 과금 있음, Premium SSD는 없음

# Part 2. Inform (보이기)

## 1. 비용 관리 체계: 관리그룹·청구계층·태그

### 관리그룹(Management Group)

- **정의**: 여러 구독을 계층으로 묶어 정책(Policy)·RBAC(역할 기반 권한 제어)·비용 예산/알림을
  그룹 단위로 관리하는 논리적 컨테이너
- 관리그룹별 비용 가시화는 **EA 계약에서만 지원**
- **계층 구조 예시(최대 6단계)**
  - Tenant Root Group — 구독 직접 배치 금지
  - └ {회사} 최상위 관리그룹
    - Platform — 공유 인프라(네트워크·보안·로깅)
    - Landing Zones — 워크로드(Corp / Online)
    - Sandbox — 실험·PoC(정책 격리)
    - Decommissioned — 폐기 예정 격리
- **작성 가이드**
  1. 깊이 3 ~ 4단계 이내(최대 6단계 가능하나 깊을수록 정책 상속 추적 난이도 상승)
  2. 루트에 구독 직접 배치 금지(루트 정책은 전사 영향 — 사고 시 전면 마비 위험)
  3. 관리 주체·정책 요구로 분리(환경 Prod/Dev이 아닌 같은 정책·같은 팀끼리 묶기)
  4. FinOps 태그와 정렬(관리그룹 = 정책 경계, 태그 = 세밀 배분, 혼동 금지)
  5. Sandbox 격리 필수(프로덕션 가드레일에서 분리하되 예산 알림은 필수)

### 세 도구의 역할 구분

- 관리그룹·청구계층·태그는 목적이 다른 **별개의 도구 3개**이며, 셋이 모두 만나는 지점이 '구독(Subscription)'임

| 축 | 역할 | 커버리지 | 정산 기준 |
|----|------|----------|-----------|
| 관리그룹 | 거버넌스(규칙) | 100% | 보조 |
| 청구계층 | 정산(청구서) | 100% | 주력 |
| 태그 | 세밀 배분 | 빈틈 있음 | 단독 불가 |

- **EA vs MCA 차이**(관리그룹·태그는 동일, 청구계층만 다름)
  - EA: 청구 계정 → 부서 → 등록 계정 → 구독. **관리그룹 스코프 비용 조회 지원**
  - MCA: 청구 계정 → 청구 프로파일 → 청구서 섹션 → 구독. **관리그룹 스코프 미지원**(청구 프로파일·섹션으로 조회)
  - 출처: MS Learn — 관리그룹 스코프 비용 분석은 EA만 지원, MCA는 현재 미지원

### 합치면 안 되는 3원칙

1. 정산 기준은 '커버리지 100%' 축(청구계층)으로 — 태그로 청구서 만들면 미태깅 잔액이 총액과 불일치
2. 관리그룹 ↔ 청구계층 억지 1:1 금지 — 느슨하게 연결(조직 개편 시 양쪽 동시 붕괴 방지)
3. 관리그룹 ≠ 조직도 — 규칙 경계로만 사용, 부서·조직 정보는 태그에 담기

### 실행 시 지킬 4가지(BU 매핑)

- 안 바뀌는 BU는 청구계층(정산 앵커), 자주 바뀌는 팀은 태그(유동)로 관리
  1. MCA의 BU = Billing Profile 의미 합의(통합 청구서면 BU = Invoice Section)
  2. 하위 계층(EA Account, MCA Invoice section)은 안정축(환경·제품군 등)에 매핑
  3. BU 태그를 항상 병행 부여(단일 태그 리포트로 통일, 청구계층은 정산·감사 앵커)
  4. BU도 '코드 → BU명' 매핑표로 관리(배치가 바뀌어도 리포트가 안 깨짐)

### 태그가 못 잡는 3종(주의)

1. 미태깅 리소스 — 사람이 깜빡·오타 → 커버리지 100% 불가
2. 소급 미적용 — 태그는 붙인 이후만 반영, 과거 비용 제외
3. 예약·SP·마켓플레이스 — 구매 항목은 태그 자체가 안 붙음

> 이 3종은 구독·리소스 그룹 기준 '배분 규칙'으로 100% 보완함

- **시작 로드맵**: Crawl(필수 태그 4종·총액 정합·Showback) → Walk(Policy 태그 강제·미태깅 리포트·커버리지 KPI)
  → Run(Chargeback·예약/SP amortization·단위 경제 분석)

> 관련 실습: `hands-on/01.management-group.md`, `hands-on/02.billing-profile_invoice-section.md`

## 2. 태그·명명 거버넌스

- **목적**: 비용을 누가·왜·어디에 사용했는지 추적. 태그 미적용 리소스 비율 5% 이하 유지가 목표

| 구분 | 태그 키 | 예시 값 | 용도 |
|------|---------|---------|------|
| 비즈니스 | team | 개발팀, 마케팅팀 | 부서별 비용 추적 |
| 비즈니스 | project | order-service | 프로젝트별 비용 할당 |
| 비즈니스 | CostCenter | CC-1234 | 비용 센터 연계 |
| 기술 | env | prod, staging, dev | 환경별 비용 분리 |
| 기술 | Owner | team-platform | 비용 책임자 식별 |

- **태그 거버넌스**
  1. 태그 정책 강제(Tag Policy) — IaC 배포 시 필수 태그 미적용이면 배포 차단
  2. 미태깅 리소스 탐지 — 주간 스캔으로 리포트 생성
  3. 태그 표준화 — 네이밍 컨벤션 문서화·자동 검증

# Part 3. Optimize (줄이기)

## 1. Right-sizing: 적정 VM SKU 판단법

- **WHY**: 과대 프로비저닝 = 낭비, 과소 프로비저닝 = 장애 리스크. 감이 아닌 측정된 사용량이 근거
- **HOW(4단계)**
  1. 메트릭 관찰(Azure Monitor) — 1 ~ 2주, P95 기준, CPU·메모리·디스크 IO·네트워크 병행(피크 포함)
  2. Azure Advisor 확인 — 참고치. Right-size 권고는 CPU·아웃바운드 네트워크만 반영, **메모리 미반영**
  3. 3대안 비교 — 월 비용·헤더룸(실제 사용량과 할당 용량 차이, 20 ~ 30%↑)·리스크·다운타임을 동일 기준 비교
  4. 선정·적용·재검증 — 저위험 시간대 적용 → 1 ~ 2주 재관찰 → 점진적 축소 반복
- **핵심 원칙**: 헤더룸 20 ~ 30% 이상 확보 후보만 최종 선정, Advisor는 참고치일 뿐 자동 적용 금지
- **흔한 함정**: 단발 스냅샷 판단, 평균(Avg)만 보기, CPU만 보기, 다운타임 미고려, 한 번에 과도 축소

- **결과물 예시**(관찰 P95: CPU 20%, 메모리 20% 기준)

| 구분 | SKU | vCPU/메모리 | 헤더룸 | 리스크 |
|------|-----|-------------|--------|--------|
| 현재 | D4s_v6 | 4 / 16GB | 80%/80% 과잉 | 없음 |
| 후보 A(선정) | D2s_v6 | 2 / 8GB | 60%/60% 확보 | 재부팅 1회 |
| 후보 B | B2ms | 2 / 8GB | 메모리 60%·CPU 버스트 | 크레딧 소진 시 성능 저하 |
| 후보 C | B2s | 2 / 4GB | 60%/메모리 20% | 재부팅+메모리 부족+크레딧 소진 |

> 선정 로직: ① 헤더룸 20 ~ 30%↑ 통과 → ② 다운타임 SLA 내 허용 → ③ 동일 기준 최소 비용 선정
> → ④ 축소 적용 후 1 ~ 2주 재관찰·재검증, 문제 시 즉시 롤백

## 2. Right-sizing: Disk SKU 비용

- 같은 32GiB라도 SKU 구조에 따라 과금 방식이 다름 — 고정형은 계층 단가, 구성형은 용량·성능 분리 과금
- **디스크 계층(Disk tier)**: 미리 정의된 용량·기본 성능을 묶은 단계.
  Standard HDD=S · Standard SSD=E · Premium SSD=P + 숫자(예: 32GiB → S4/E4/P4). 프로비전 계층 기준 과금
  - Blob의 Hot·Cool·Archive 액세스 계층이나 LRS·ZRS 중복성 옵션과는 다른 개념

- **고정 계층형 SKU (32GiB 비교)**

| Disk SKU | 계층 | 월 소매가 | 과금 특징 |
|----------|------|-----------|-----------|
| Standard HDD LRS | S4 | $1.536 | 디스크 작업 10,000건당 $0.0005 별도 |
| Standard SSD LRS | E4 | $2.40 | 트랜잭션 비용 별도 |
| Premium SSD LRS | P4 | $5.2795 | 계층에 포함된 성능 제공 |

- **구성형 SKU (32GiB 용량 비용)**

| Disk SKU | 730시간 기준 월 환산 | 별도 과금 |
|----------|----------------------|-----------|
| Premium SSD v2 LRS | 약 $2.92 | 기본 3,000 IOPS·125MB/s 초과 성능 |
| Ultra Disk LRS | 약 $5.1626 | 프로비전 IOPS 및 처리량 |

> - 구성형은 용량·IOPS·처리량을 독립 설정 → 고정 계층형과 직접 비교 어려움
> - 초과 성능 과금(Premium SSD v2): 3,000 IOPS 초과분 IOPS당 시간당 $0.000008, 125MB/s 초과분 MB/s당 시간당 $0.000063
> - Ultra Disk: GiB당 시간당 $0.000221 + 프로비전 IOPS당 $0.000092 + 프로비전 MB/s당 $0.00049
> - 2026-07-22 Azure Retail Prices API 조회값(Korea Central·LRS·USD 소매가, 계약 할인·세금·환율 미포함)

## 3. Right-sizing: Blob Tiering

- Blob Tiering은 저장한 파일을 사용 빈도에 따라 서로 다른 보관함(계층)으로 옮기는 방식

| 항목 | Hot | Cool | Cold | Archive |
|------|-----|------|------|---------|
| 저장 단가(1GB/월) | $0.0192 | $0.011 | $0.0045 | $0.002 |
| 읽기 작업(10K) | $0.004 | $0.01 | $0.10 | $5.50 |
| 최소 보존 기간 | 없음 | 30일 | 90일 | 180일 |
| 온라인 여부 | O | O | O | X(복원 필요) |

> Korea Central·USD·LRS·Block Blob(GPv2)·2026-07-19 조회. 실제 값은 Pricing Calculator 확인

- **FinOps 체크리스트 & 함정**
  1. 너무 빨리 지우거나 옮기면 위약금 발생 — Hot 외 계층은 일정 기간 보관 조건으로 저렴(90일 조건 데이터 10일 삭제 시 추가 비용)
  2. 저렴한 계층이 항상 저렴하지 않음 — 자주 접근하면 Cool/Cold/Archive가 오히려 더 비쌈(보관료 + 읽기·검색·복원 비교)
  3. 작은 파일이 많으면 처리 비용 증가 — 계층 이동·읽기는 건당 트랜잭션 과금, 소형 파일 다수는 절감분 상쇄
  4. 자동 이동 규칙 기준일 오해 — 기본은 최종 수정일(≠최종 접근일), Archive 사용 시 online 복원 필요, 실행은 하루 1회
  5. 실측 우선 — 사용 현황 확인 → 일부 데이터 시험 → 비용 검증 → 전체 확대

> 계층이 차가워질수록 저장은 싸지나 읽기·검색 단가는 급등.
> Archive 읽기($5.50/10K)는 Hot 읽기($0.004/10K)의 약 1,375배 수준.
> 저장 단가만 내리는 것은 함정 — 접근 빈도 × 트랜잭션·검색·최소 보존을 함께 봐야 실제 절감됨.
> 관련 실습: `hands-on/11-1.blob-tiering.md`, `hands-on/11-2.smart-tiering.md`

## 4. Right-sizing: Spot 혼합 전략

- 미사용 용량을 큰 폭 할인가로 쓰되 언제든 축출됨 — 표준 VM 베이스라인 위에 Spot을 얹어 가용성·절감 동시 확보
- **WHY**: Spot VM은 Azure 미사용 용량을 큰 폭 할인 제공(SLA·고가용성 보장 없음),
  축출(evict) 시 최소 30초 사전 통지(best-effort), 중단 허용·무상태 워크로드에만 적용
- **HOW**
  - Spot Priority Mix: 단일 VMSS에서 표준 + Spot 혼합, **Flexible 오케스트레이션 모드 필수**
  - Base VM count = 항상 유지되는 표준 VM 최소 가용량 / Instance distribution(pct) = 초과분 중 표준 비율
  - autoscale 사용 시 축출 정책은 **Delete 권장**(Deallocate는 quota·디스크 과금 누수)
- **WHAT**
  - 예시: base=10, pct=50인 VMSS를 30대로 확장 → 표준 20대 + Spot 10대
  - 적합: 배치 처리·데이터 분석·dev/test / 부적합: 프로덕션 DB·세션·결제 트랜잭션
  - 안티패턴: stateful을 Spot 배치 금지, 단일 Spot 의존 금지, autoscale + Deallocate 조합 금지

| 구분 | Deallocate(기본) | Delete |
|------|------------------|--------|
| 재배포 | 가능(성공 보장 없음) | 불가(새로 생성) |
| 디스크 과금 | 계속 과금 | 없음 |
| 권장 상황 | 단발 Spot VM | VMSS autoscale |

## 5. 스케일링 정책 최적화 (HPA/VPA/노드)

- **HPA(Horizontal Pod Autoscaler) 비용 관점**

| 설정 항목 | 비용 영향 |
|-----------|-----------|
| CPU 임계값(50% vs 70%) | 낮을수록 빠른 스케일아웃 → Pod 증가 → 비용 상승 |
| 최소 Pod 수(minReplicas) | 높을수록 유휴 비용 발생 |
| 최대 Pod 수(maxReplicas) | 피크 시 비용 상한선 결정 |
| 스케일다운 지연(cooldown) | 짧으면 불안정, 길면 불필요 비용 지속 |

- **VPA(Vertical Pod Autoscaler)**: Request/Limit 자동 조정으로 Right-sizing 자동화, Over-provisioning 방지
  - 주의: VPA + HPA 동시 사용 시 HPA는 CPU 기준, VPA는 메모리 기준 권장.
    CPU만 기준 시 VPA가 CPU 증가 → 사용량 감소 → HPA가 Pod 축소 → **Flapping 현상** 발생
- **Cluster Autoscaler / Karpenter**: 노드 수준 스케일링이 전체 비용에 직접 영향.
  빈 패킹(Bin Packing)으로 Pod를 최소 노드에 밀집·유휴 노드 제거. Karpenter는 워크로드 맞춤 최적 인스턴스 자동 선택

- **비용 트레이드오프**

| 정책 | 보수적(과잉) | 공격적(최소) |
|------|--------------|--------------|
| 비용 | 높음(불필요 노드 유지) | 낮음 |
| 성능 위험 | 낮음 | 높음(피크 시 저하) |
| 권장 | 프로덕션 핵심 서비스 | 개발/테스트 환경 |

> Best Practice: 스케일링 비용 시뮬레이션으로 예상 트래픽 기반 월간 비용 사전 추정.
> 관련 실습: `hands-on/07-1.vmss-autoscaling.md`, `hands-on/07-2.add-vm-to-vmss.md`
> (오케스트레이션 모드를 '유연한'으로 설정하면 기존 VM을 VMSS에 추가 가능, 모니터링 주기 5분까지 축소 가능)

## 6. 약정 할인 의사결정

| 구분 | Reserved Instances(RI) | Savings Plans(SP) | Spot Instances |
|------|------------------------|-------------------|----------------|
| 할인율 | 최대 72% | 최대 72% | 최대 90% |
| 약정 기간 | 1년/3년 | 1년/3년 | 없음 |
| 유연성 | 낮음(특정 인스턴스 고정) | 높음(인스턴스 변경 가능) | 매우 높음 |
| 중단 위험 | 없음 | 없음 | 있음(회수 가능) |
| 적합 워크로드 | 24/7 상시(DB·기간계) | 인스턴스 변경 빈번한 서비스 | 배치·CI/CD·데이터 분석 |

- **의사결정 매트릭스**
  - 안정적 + 장기 + 인스턴스 고정 → RI
  - 안정적 + 장기 + 인스턴스 유동적 → Savings Plans
  - 중단 허용 + 단기/탄력적 → Spot
  - 불확실 + 초기 탐색 → On-Demand(약정 없이 필요 시 구매)
- **혼합 전략(Blended Approach)**: 베이스라인은 RI/SP(안정적 기반), 피크 대응은 On-Demand,
  배치·비정기는 Spot으로 비용 극소화
- 약정 구매 전 최소 3 ~ 6개월 사용 데이터 분석 필수

### [참고] Amortized(분할상환) 방식

- 선불 비용을 사용 기간에 걸쳐 균등 배분하여 공정한 비용 할당 실현
  - 비유: 헬스장 연회원권 120만원 결제 → 결제 달에 120만원이 아니라 매달 10만원씩 쓴 것으로 장부 기록

| 방식 | 1월 | 2월 | 3 ~ 11월(각) | 12월 | 연간 합계 |
|------|-----|-----|--------------|------|-----------|
| Cash(현금주의) | $1,200 | $0 | $0 | $0 | $1,200 |
| Amortized(발생주의) | $100 | $100 | $100 | $100 | $1,200 |

> Amortized가 필요한 이유: 실제 지출(Cash)은 결제 당월 폭증·나머지 0원으로 추이 분석이 왜곡됨.
> Amortized는 계약 기간에 균등 분배하여 혜택받은 서비스/부서에 정확히 할당, 월별 추이가 안정적·예측 가능.
> → 공정한 Chargeback 실현 및 비즈니스 가치 정확 측정

### 할인 대상 판별

- 상시·안정 부하만 약정 대상 — Right-size 먼저, 컴퓨팅/용량만 커버, Advisor·Cost Management 권장 확인 후 단계적 구매

| 판별 축 | 대상(약정 O) | 비대상(약정 X) |
|---------|--------------|----------------|
| 상시성 | 24/7 상시, 인스턴스·리전 변경 없음 | 켜졌다 꺼졌다·간헐·단기 실험 |
| 비용 성격 | 컴퓨팅/용량(또는 SUSE·RedHat 플랜) | 일반 SW 라이선스·네트워크·스토리지 트랜잭션 |
| 권장 신호 | Advisor/CM 권장 + 예상 절감 > 0 | 권장 미표시(관측 부족·간헐) |

- **RI vs SP 핵심 차이**
  - 약정 방식: RI는 인스턴스/패밀리 + 특정 리전(1·3년) / SP는 시간당 $ 고정 지출(1·3년)
  - 리전 종속: RI는 특정 리전에만 적용(이동 시 미적용) / SP는 참여 서비스·전 리전 자동 적용
  - 취소·환불: RI는 교환 가능·환불(12개월 $50,000 한도) / SP는 취소·환불 불가
- **핵심 함정 2가지**
  1. 리전·패밀리 불일치(RI) — 다른 리전·패밀리로 구매 시 할인 전혀 미적용
  2. 7·30·60일 lookback 과소 추천 — 신규·변동 워크로드는 관측 부족으로 권장 축소·미표시, 며칠 관측 후 재확인
- **구매 원칙(실행 순서)**
  1. Right-size 먼저 — 할인은 요율만 낮추고 낭비는 제거 못 함
  2. 최적화 순서: 저활용 예약 교환 → SP 전환 → 신규 예약 → 신규 SP
  3. 권장값에 근접 구매 — 과다·과소 구매 모두 절감률 저하
  4. RI·SP 동시 구매 금지 — 하나 구매 후 3일 대기(권장 재계산 반영)

> 관련 실습: `hands-on/14.ri-sp-spot.md`, `hands-on/08.advisor-ri-sp.md`

# Part 4. Operate (체계화)

## 1. FinOps 자동화

- 탐지 → 권고 → 승인 → 적용 → 검증 파이프라인 구축

- **FinOps 도구 생태계**

| 구분 | 도구 | 특징 |
|------|------|------|
| CSP 네이티브 | AWS Cost Explorer, Azure Cost Mgmt, GCP Billing | 무료, 기본 기능 |
| 써드파티 | Apptio Cloudability, CloudHealth, Kubecost | 멀티클라우드, 고급 분석 |
| 오픈소스 | OpenCost, Infracost | Kubernetes 특화, IaC 비용 예측 |
| 국내 | 삼성SDS Cloud inOne | 하이브리드/멀티클라우드 통합 관리 |

- **FOCUS(FinOps Open Cost & Usage Specification)**
  - FinOps Foundation 주도의 멀티클라우드 비용 데이터 표준
  - CSP별 상이한 청구 데이터 포맷을 통일된 스키마로 정규화 → 멀티클라우드 비용 분석 단순화
  - 57% 조직이 FOCUS 도입 계획(State of FinOps 2025)

- **비용 최적화 자동화 사례**

| 자동화 유형 | 구현 방법 | 절감 효과 |
|-------------|-----------|-----------|
| 미사용 리소스 정리 | Lambda/스크립트로 주기 스캔·삭제 | 즉시 절감 |
| 스케줄 기반 시작/중지 | 비운영 시간대 개발 환경 자동 중지 | 40 ~ 60% |
| 태그 정책 강제 | IaC 파이프라인에 태그 검증 통합 | 가시성 확보 |
| Right-sizing 권고 | Compute Optimizer + 자동 적용 | 10 ~ 30% |

- **비용 최적화 파이프라인**: 1) 탐지(Detect) → 2) 권고(Recommend) → 3) 승인(Approve) → 4) 적용(Apply) → 5) 검증(Verify)
- **Shift-Left FinOps**: IaC(Terraform·CloudFormation)와 비용 정책 통합,
  Infracost로 terraform plan 시 예상 비용 변화 자동 표시, PR 리뷰에 비용 영향 분석 포함(배포 전 비용 인식)

## 2. 단위 경제(Unit Economics)와 KPI 설계

- 총비용 증가 ≠ FinOps 실패 — 비즈니스 성장을 수반하는 비용 증가는 건강한 성장
- 핵심 질문: **"트래픽 1건 처리 비용이 감소하고 있는가?"**(우아한형제들: 클라우드 비용 ÷ 트래픽 수)

- **활성화/도입 메트릭**

| KPI | 설명 | 목표 예시 |
|-----|------|-----------|
| 태깅 커버리지 | 태그 적용된 리소스 비율 | 95% 이상 |
| 미할당 비용 비율 | 부서에 할당되지 않은 비용 | 5% 이하 |
| 청구 데이터 최신성 | 비용 데이터 반영 지연 시간 | 24시간 이내 |
| RI/SP 커버리지 | 약정 할인이 적용된 비율 | 70% 이상 |

- **성과 메트릭**

| KPI | 설명 | 산업별 예시 |
|-----|------|-------------|
| 단위당 비용 | 비즈니스 단위당 클라우드 비용 | 이커머스: 주문 1건당 비용 |
| 예측 정확도 | 예산 대비 실제 차이 | MAE/MAPE 10% 이내 |
| RI/SP 활용률 | 구매 약정의 실제 사용 비율 | 80% 이상 |
| 회수된 낭비 | 최적화로 절감한 금액 | 월간 추적 |

- **단위 경제 예시**: 이커머스(주문 1건 = 비용÷주문수), 물류(배송 1건 = 비용÷배송수),
  SaaS(MAU당 = 비용÷MAU), AI 서비스(추론 1건 = 비용÷추론 요청수)

> MAE(Mean Absolute Error): 예측값과 실제값 차이 평균 / MAPE(Mean Absolute Percentage Error): 그 차이의 평균 비율

## 3. 협업 문화와 거버넌스 체계

- **Cost-Aware Engineering 문화**
  - 가시성 제공: 엔지니어에게 서비스 비용 대시보드 제공
  - 인센티브 설계: 비용 절감 성과를 팀 평가에 반영
  - 교육 프로그램: FinOps 기본 교육을 온보딩에 포함
  - 셀프서비스: 엔지니어가 직접 비용 분석·최적화 가능한 환경

- **FinOps 팀 핵심 역할**

| 역할 | 책임 | 소속 |
|------|------|------|
| FinOps Practitioner | 전사 비용 거버넌스 운영, 정책 수립, 도구 관리 | FinOps 팀(중앙) |
| 엔지니어 | 서비스별 비용 최적화 실행, 아키텍처 개선 | 개발팀(분산) |
| 재무 담당 | 예산 관리, 비용 보고, ROI 분석 | 재무팀 |
| Product Owner | 비즈니스 가치 기반 우선순위 결정 | 사업팀 |
| 경영진 스폰서 | 전략적 의사결정, 조직 변화 지원 | C-Level |

- **비용 리뷰 운영**

| 주기 | 참석자 | 내용 |
|------|--------|------|
| 주간 | FinOps 팀 + 엔지니어 | 이상 비용 리뷰, 최적화 액션 추적 |
| 월간 | + 재무·사업 담당 | 예산 대비 실적, KPI 리뷰, 트렌드 분석 |
| 분기 | + 경영진 | 전략 정렬, 투자 의사결정, 성숙도 평가 |

## [별첨] FinOps 성숙도 모델 중요 변화

- 단일 점수(Single Score) 모델에서 다차원 입체 진단 모델로의 패러다임 전환
  - **과거(AS-IS)**: 전사 단일 점수 측정(평균의 함정)
  - **2025 업데이트(TO-BE)**: Ownership & Capability 기반 입체 진단, 3 ~ 6개월 주기 평가

- **Ownership(비용 할당) — 조직·환경 단위 세분화**
  - 전체 회사가 아닌 실제 운영 주체(서비스팀·플랫폼팀)나 환경(Prod/Dev) 단위로 측정 범위 세분화
  - 우수 팀의 Best Practice를 발굴하고 역량 부족 팀을 정확히 찾아 집중 지원

- **Capability(핵심 역량) — 20여 개 기능적 세부 역량 평가**
  - 비용 할당·데이터 분석·이상 탐지·약정 관리 등 실무 기능별 숙련도 점검
  - "데이터 분석은 뛰어나나 최적화 실행력이 부족함" 같은 명확한 진단으로 팀별 시급한 개선 포인트 도출

> 전사 평균 점수에 집착하지 않기. 클라우드 비용이 가장 높은 부서를 우선 타겟팅하여,
> 비용 가시성 확보(Capability)부터 단기간(3 ~ 6개월) 내 달성하는 것이 핵심.
