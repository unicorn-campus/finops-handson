# 사용량(Usage) 전용 추이 분석법 — FOCUS ConsumedQuantity (Cost summary 리포트)

> 대상 리포트: FinOps Toolkit **Cost summary**(Power BI) > **Usage analysis** 페이지  
> 대상 데이터: **FOCUS 1.0** 형식 Azure Cost Management Export(스토리지 연결 기준)  
> 목적: 비용 추이에서 **사용량(수요, demand)** 을 분리해 "무엇이 실제로 늘었는가"를 규명함  
> ※ 본 문서의 클릭 경로·측정값·컬럼명은 리포지토리 실물(리포트 Layout · FOCUS 스키마 메타데이터 ·  
> EA-Cost-FOCUS_1.0 실측 집계)로 검증한 범위에서만 기술함. 확인 불가 항목은 "환경별 상이/미확인"으로 표기함.

---

## 0. 왜 '비용'이 아닌 '사용량'을 보는가 (WHY)

> **Fact.** 클라우드 청구비용은 세 요소의 곱으로 분해됨:  
> **비용 = 사용량(ConsumedQuantity) × 단가(unit price) × 할인·약정·환율.**  
> 따라서 비용 추이 하나만으로는 "실제로 더 썼는지" 아니면 "단가·약정·환율이 바뀌었는지"를 구분 불가함.  
> 사용량(`ConsumedQuantity`)을 비용에서 분리하면 가격 노이즈가 제거되어 **수요(demand)의 순수 추세**가 드러남.  
> (FOCUS는 사용량을 `ConsumedQuantity`·`ConsumedUnit`로 표준화함 —  
> 출처: [MS Learn: What is FOCUS](https://learn.microsoft.com/en-us/cloud-computing/finops/focus/what-is-focus),  
> [FOCUS Spec](https://focus.finops.org/))

**비유.** 매달 전기요금 그래프만 보면 "가전을 더 돌렸는지" "요금 단가가 올랐는지" 알 수 없음.  
사용 **kWh 그래프**를 따로 봐야 소비 자체가 늘었는지 판별 가능 — 클라우드의 `ConsumedQuantity`가 바로 이 kWh에 해당함.

**이 분리가 FinOps에서 하는 일.** 사용량 전용 추이는 그 자체로 끝이 아니라 **여러 역량의 입력값**이 됨 —  
예측(Forecasting)은 가격을 뺀 순수 수요 추세를 필요로 하고,  
이상탐지(Anomaly Management)는 단가·환율 변동에 의한 오탐을 걸러내려면 사용량 급변을 봐야 하며,  
단위경제(Unit Economics)는 "단위당 비용·가치"의 분모(사용량 또는 비즈니스 트랜잭션)를 사용량 데이터에서 얻음.

> **Opinion.** 즉 사용량 분리는 "숫자 하나 더 보는 것"이 아니라 **비용 변동의 원인을 사용량↔단가로 귀인(attribution)** 하게  
> 해주는 Inform 단계의 진단 렌즈임. 원인을 알아야 처방(Optimize)이 맞아떨어짐.  
> (FinOps 원칙 "의사결정은 클라우드의 비즈니스 가치로 이끈다" 정합 — 출처: [FinOps Principles](https://www.finops.org/framework/principles/))

## 1. 사실: FOCUS의 '사용량' 컬럼 이해

### 1.1 버전별 컬럼 명칭 (중요: rename 이력)

FOCUS 스펙 버전에 따라 사용량 컬럼 **이름 자체가 다름**. 리포트가 어떤 버전 Export를 읽는지 먼저 확인해야 함.

| FOCUS/Export 버전 | 실제 사용량 컬럼 | 사용 단위 | 가격기준 수량 | 단가 단위 |
|---|---|---|---|---|
| FOCUS 1.0-preview (v1) | `UsageQuantity` | (ConsumedUnit **없음**) | `PricingQuantity` | `PricingUnit` |
| **FOCUS 1.0 정식** (2024-06-20) | `ConsumedQuantity` | `ConsumedUnit` | `PricingQuantity` | `PricingUnit` |
| FOCUS 1.2-preview | `ConsumedQuantity` | `ConsumedUnit` | `PricingQuantity` | `PricingUnit` |
| Azure 레거시 Actual/Amortized | `Quantity` | `UnitOfMeasure` | (해당 없음) | (해당 없음) |

- **핵심**: FOCUS 1.0 정식에서 `UsageQuantity` → **`ConsumedQuantity`로 개명**됨. 문서·측정값에서 옛 이름을 쓰면 매핑 실패.  
- 레거시(비-FOCUS) Export는 컬럼명이 아예 `Quantity`·`UnitOfMeasure`라 Cost summary(FOCUS) 스키마와 호환되지 않음.  
- 본 가이드는 **FOCUS 1.0 정식** 기준으로 `ConsumedQuantity`·`ConsumedUnit`을 사용함.

### 1.2 ConsumedQuantity vs PricingQuantity (혼동 주의)

FOCUS 1.0 공식 정의(FocusCost_1.0.json 원문 기준) 요지는 다음과 같음.

| 컬럼 | 성격 | 정의 요지 | 사용량 추세 분석에 |
|---|---|---|---|
| `ConsumedQuantity` (Decimal) | **실제 사용된 양** | "Consumed Unit 기준으로, 리소스·서비스가 사용한 SKU의 볼륨" | **사용(수요)** — 이걸 씀 |
| `ConsumedUnit` | 사용 측정 단위 | 공급자가 SKU 사용량을 측정하는 단위 | 단위 고정 기준값 |
| `PricingQuantity` (Decimal) | **가격 산정 기준 수량** | "Pricing Unit 기준으로 사용·구매된 볼륨"(블록 프라이싱 등 단가 규칙은 `PricingUnit` 정의에 근거) | 청구 검증용 — 수요와 다름 |
| `PricingUnit` | 단가 산정 단위 | 단가(EffectiveUnitPrice)에 곱해지는 단위 | — |

- **원칙**: 사용량(수요) 추세는 반드시 **`ConsumedQuantity`** 로 봄. `PricingQuantity`는 블록 프라이싱·구매 단위 규칙이  
  반영된 "청구용 수량"이라 실제 사용량과 어긋날 수 있음(§7 실측에서 약 57% 불일치 확인).  
- ⚠️ **이 페이지 비주얼에는 `PricingQuantity`가 없음**(§3 참조). 위 표는 개념 구분용이며, Usage analysis 페이지의 표는  
  `ConsumedQuantity`와 `x_PricingBlockSize`(블록 크기)를 노출함. `PricingQuantity` 자체 컬럼은 이 페이지에 배치돼 있지 않음.

## 3. Usage analysis 페이지 구조 이해 (실제 비주얼)

아래는 리포트 Layout에서 확인된 **실제 구성**임. **바인딩 필드**는 Layout에서 추출한 확정값(권위 기준)이며,  
**표시 라벨**은 화면에 나타나는 명칭임 — 굵은 `"Unit"`·`"Meter hierarchy"`는 페이지에 표기되는 라벨(dept 05 예시에서도 동일 확인),  
괄호 항목은 저자가 붙인 설명용 명칭임(해당 슬라이서에 별도 캡션이 저장돼 있지 않음).

### 3.1 슬라이서(필터) — 좌측 패널

| 표시 라벨 | 바인딩 필드(Layout 확정) | 용도 |
|---|---|---|
| **"Unit"** | `ConsumedUnit` | **사용량 단위 고정**(가장 중요, §6-①) |
| (통화) | `BillingCurrency` | 통화 필터 |
| (구독/리소스그룹) | `SubAccountName` + `x_ResourceGroupName` | 범위 좁히기 |
| (약정 유형/이름) | `CommitmentDiscountType` + `CommitmentDiscountName` | 예약·SP 관련 필터 |
| (서비스) | `ServiceCategory` + `ServiceName` | 서비스 필터 |
| (기간) | `ChargePeriodStart` | 날짜 범위 |
| **"Meter hierarchy"** | `x_SkuMeterCategory` + `x_SkuMeterSubcategory` + `SkuMeter` | 미터 계층 드릴다운 |

### 3.2 카드(KPI) — 상단

| 카드 | 측정값 | 의미 |
|---|---|---|
| 비용 | `Sum(EffectiveCost)` | 실효 비용 합계 |
| ESR | `x_EffectiveSavingsRate` | 실효 절감률 |
| 절감액 | `Sum(x_TotalSavings)` | 총 절감액 |

> 참고: 카드 3종은 모두 **비용·절감** 지표임. **사용량 KPI 카드는 이 페이지에 없음**(§4에서 직접 만들거나 콤보차트로 읽음).

### 3.3 콤보차트 (line + stacked column) — 핵심 시각화

| 요소 | 바인딩 | 비고 |
|---|---|---|
| X축 | `ChargePeriodStart` | 시간축(일/월 granularity는 필드 계층으로 조정) |
| **막대(Y축)** | **`Sum(ConsumedQuantity)`** | ← **사용량**. 우리가 볼 대상 |
| 선(보조 Y축) | `Sum(EffectiveCost)` | 비용. **분리 분석 시 무시·제거 대상** |
| 범례(계열) | `x_SkuMeterCategory` | 미터 카테고리별 누적 |

- **이 콤보차트가 곧 "사용량-비용 오버레이"** 임. 사용량(막대)과 비용(선)이 한 차트에 겹쳐 있음.  
- 본 가이드의 새 가치는 이 오버레이에서 **비용 선을 걷어내고 막대(사용량)만 남겨** 추세를 읽는 것(§4).

### 3.4 표(table) — 하단 상세

| 컬럼(측정값) | 의미 |
|---|---|
| `ChargeDescription` | 청구 항목 설명 |
| `Sum(ConsumedQuantity)` | **사용량 합계** |
| `ConsumedUnit` | 사용 단위 |
| `Sum(x_PricingBlockSize)` | 가격 블록 크기 합 |
| `Sum(x_EffectiveUnitPrice)` | 실효 단가 합 |
| `Sum(EffectiveCost)` | 실효 비용 합 |

> 이 표는 "사용량 × 블록크기 × 단가 = 비용" 분해를 한 행에서 검증하기 좋음(§4.3 동인 귀속에 활용).

## 4. 사용량 전용 뷰 만들기 — 실전 절차

두 경로를 제시함. **읽기 전용(4.1)** 은 뷰어 권한만으로 즉시 수행, **편집모드(4.2)** 는 재사용 가능한 전용 시각화를 남김.

### 4.1 경로 A — 읽기 전용(기존 콤보차트에서 비용 선 "무시하며" 읽기)

수정 권한 없이 오버레이 차트에서 사용량만 추세로 읽는 방법임.

1. **단위 1개 고정**: 좌측 **Unit 슬라이서(`ConsumedUnit`)** 에서 단위 하나만 선택(예: `Hours`).  
   → 단위 혼합 합산 방지(§6-① 필수). 이 고정 없이는 막대 높이가 무의미함.  
2. **비용 선 무시**: 콤보차트에서 **선(EffectiveCost, 보조 Y축)** 은 시선에서 배제하고 **막대(ConsumedQuantity)** 만 읽음.  
   보조 Y축 눈금(우측)은 비용 스케일이므로 참고하지 않음.  
3. **날짜 범위 확장**: **기간 슬라이서(`ChargePeriodStart`)** 를 분석 대상 전 구간으로 넓혀 추세선을 길게 확보함.  
4. **granularity 조정**: (축에 날짜 계층·드릴 컨트롤이 활성인 경우) X축(`ChargePeriodStart`)에서 **일 ↔ 월** 을 전환하며 추세를 봄  
   (일 단위: 급증 격리에 유리 / 월 단위: 장기 추세에 유리). ※ 콤보차트 X축은 단일 날짜 필드 바인딩이라 드릴은 환경 의존.  
5. **범위 좁히기**: 필요 시 서비스·구독·리소스그룹 슬라이서로 대상을 한정한 뒤 다시 막대만 읽음.  

> 한계: 읽기 전용에서는 비용 선을 **화면에서 제거할 수 없음**(무시만 가능). 시각적 간섭을 완전히 없애려면 4.2로.

### 4.2 경로 B — 편집모드(사용량 전용 시각화 신설·복제)

Power BI Desktop 편집에서 **비용 선이 없는** 순수 사용량 차트를 만듦. 두 방식 중 택1.

**방식 ① 기존 콤보차트 복제 후 비용 선 제거**

1. Power BI Desktop에서 **보기 > 리포트 편집** 진입.  
2. Usage analysis 페이지의 **콤보차트 선택 > 복사(Ctrl+C) > 붙여넣기(Ctrl+V)**.  
3. 복제본 선택 상태에서 **시각화 창 > 필드 영역**의 **"선 값(Line y-axis / 보조 Y)"에서 `Sum(EffectiveCost)` 제거**.  
   → 남는 것은 막대(`Sum(ConsumedQuantity)`)뿐. 사실상 묶은세로막대 차트가 됨.  
4. 필요 시 **시각화 유형을 "묶은 세로 막대형" 또는 "꺾은선형"** 으로 변경(비용 선이 없으니 콤보 유지 불필요).  

**방식 ② 신규 시각화 처음부터 구성(권장 — 재사용 표준 뷰)**

1. 편집 모드에서 빈 영역 클릭 후 **시각화 창 > "꺾은선형 차트"**(또는 "묶은 세로 막대형") 추가.  
2. 필드 바인딩:  

   | 시각화 슬롯 | 바인딩 필드 |
   |---|---|
   | X축 | `ChargePeriodStart` |
   | Y축(값) | `Sum(ConsumedQuantity)` |
   | 범례 | `ServiceName` **또는** `x_SkuMeterCategory` |

3. **단위 고정 연동**: 페이지의 **Unit 슬라이서(`ConsumedUnit`)** 가 이 신규 차트에도 적용되는지 확인  
   (동일 페이지면 기본 적용). 단위를 1개로 고정한 상태에서만 Y축 합산이 유효함.  
4. **비용 지표 미포함**: 이 차트에는 `EffectiveCost`·`x_TotalSavings` 등 **비용 계열을 넣지 않음**(사용량 전용 유지).  
5. **정렬·서식**: X축을 시간 오름차순 정렬, Y축 제목에 단위 명기(예: "ConsumedQuantity (Hours)")로 오독 방지.  

> ⚠️ 편집 저장은 원본 리포트 파일을 변경함. 실습에서는 **다른 이름으로 저장** 하거나 복제 페이지에서 작업 권장.

### 4.3 동인 귀속 — "어느 서비스·미터가 추세를 견인하는가"

사용량 추세가 확인되면, 그 상승·하강을 **특정 서비스·미터에 귀속**시켜야 액션이 나옴.

1. §4.1/4.2 차트의 **범례를 `ServiceName`** 으로 설정 → 서비스별 사용량 기여 분해.  
2. 특정 서비스가 두드러지면 **범례를 `x_SkuMeterCategory`(→ 필요 시 Meter hierarchy 드릴다운)** 로 전환해 미터 수준까지 좁힘.  
3. **하단 표(§3.4)** 를 병행: 같은 필터 조건에서 `ChargeDescription`별 `Sum(ConsumedQuantity)`·`ConsumedUnit`·  
   `x_EffectiveUnitPrice`를 대조해 "사용량 증가 vs 단가 변동" 중 무엇이 비용을 밀어올렸는지 분리 확인.  
4. 결론 문장 형식 예: "○월 사용량 증가는 `ServiceName=△` / `x_SkuMeterCategory=□` 의 `Hours` 사용 증가가 견인함".  

### 4.4 이상 신호 탐지

1. **급증월 격리**: X축을 **월 단위**로 두고 튀는 막대를 식별 → 해당 월만 기간 슬라이서로 **격리**.  
2. **일 단위 드릴다운**: 격리한 월에서 X축을 **일 단위**로 바꿔 급증이 **특정일 스파이크**인지 **지속 상승**인지 판별.  
3. **미터 드릴다운**: Meter hierarchy 슬라이서(`x_SkuMeterCategory` → `x_SkuMeterSubcategory` → `SkuMeter`)로  
   급증을 유발한 미터를 특정.  
4. **음수·조정 확인**: 사용량·비용에 음수 행(credit·환불·조정)이 섞이면 합계가 왜곡됨 → §6-④ 참조.

## 5. 추이 해석 렌즈 (패턴 유형 + FinOps 매핑)

> **각주.** FinOps Foundation은 라이프사이클 **단계명(Inform·Optimize·Operate)** 과 2024 개편 **도메인명(Understand·  
> Quantify·Optimize·Manage)** 을 병존시킴(버전별 명칭 상이). 본 가이드는 리포지토리 타 문서와 통일해 **Inform·Optimize** 표기를  
> 우선 쓰고, 아래 매핑 표에만 도메인·역량명을 병기함.

### 5.1 FinOps 역량·단계 매핑

> **Fact.** 아래 역량 명칭·소속 도메인은 FinOps Foundation Framework 기준(출처 링크는 각 행).

| FinOps 도메인 · 역량 | 사용량 전용 추이 분석이 이 역량에 기여하는 방식 | 1차 출처 |
|---|---|---|
| Understand Usage & Cost / **Data Ingestion** | FOCUS `ConsumedQuantity`·`ConsumedUnit`를 정규화 수집 — 분리 분석의 데이터 토대 | [data-ingestion](https://www.finops.org/framework/capabilities/data-ingestion/) |
| Understand Usage & Cost / **Reporting & Analytics** | 사용량을 비용과 분리한 시계열로 가시화해 수요 추세를 이해관계자에게 전달 | [reporting-analytics](https://www.finops.org/framework/capabilities/reporting-analytics/) |
| Understand Usage & Cost / **Anomaly Management** | 비용이 아닌 **사용량 급변**을 탐지 → 단가·환율 변동에 의한 오탐 감소 | [anomaly-management](https://www.finops.org/framework/capabilities/anomaly-management/) |
| Quantify Business Value / **Forecasting** | 가격·할인 노이즈를 제거한 **순수 수요 추세**로 미래 사용량 예측 정확도 향상 | [forecasting](https://www.finops.org/framework/capabilities/forecasting/) |
| Quantify Business Value / **Unit Economics** | 단위경제 지표(비용/사용량, 사용량/트랜잭션)의 **분모·분자** 구성요소 제공 | [unit-economics](https://www.finops.org/framework/capabilities/unit-economics/) |
| Optimize Usage & Cost / **Cloud Sustainability**(해당 시) | 사용량(리소스 소비량)은 에너지·탄소의 대리지표 → 비용과 별개로 추적 필요 | [cloud-sustainability](https://www.finops.org/framework/capabilities/cloud-sustainability/) |

### 5.2 사용량 추이 패턴 → 해석 렌즈·행동

> **Opinion (실무 휴리스틱).** 사용량 시계열 형태별 FinOps적 의미와 Inform→Optimize 브릿지 행동임.  
> **핵심 원칙**: 처방(약정·rightsizing·shutdown·회수)은 **사용량 패턴에 따라 매번 재판단**해야 함.  
> 예컨대 **간헐적 소비에 예약·약정을 걸면 오히려 낭비**가 됨(예약분을 놀림).

| 사용량 패턴 | 곡선 특징 | FinOps 해석(가설) | 다음 행동 (Optimize 브릿지) |
|---|---|---|---|
| **꾸준 (Steady)** | 평탄·낮은 변동 | 안정 상시 워크로드 | 예약·Savings Plans 적합 → Rate Optimization |
| **성장 (Growth)** | 우상향 추세 | 수요 확대 | 예측·예산 재조정 — 단 Unit Economics로 "건강한 성장"인지 확인 |
| **간헐·버스트 (Spiky)** | 불규칙 스파이크 | 배치·이벤트성 | 약정은 낭비 위험 → 온디맨드·Spot·Autoscale |
| **계단식 (Step)** | 특정 시점 레벨 점프 | 신규 배포·확장 | 이벤트 상관 확인, Anomaly와 구분 → 원인 태깅·할당 점검 |
| **소멸 (Decaying)** | 우하향·0 수렴 | 폐기·마이그레이션, 유휴 후보 | auto-shutdown·회수(reclamation) → Workload Optimization |

### 5.3 이론적 함정 (사용량 해석의 전제)

> **Fact / 이론 원칙.**  
> 1. **사용량 증가 ≠ 낭비.** 정상 수요 성장과 비효율(과대 프로비저닝)은 사용량 곡선만으로 구분 불가 —  
>    "이 사용량이 그만한 비즈니스 가치를 냈는가"는 **단위경제(Unit Economics)** 로만 판별됨  
>    (출처: [unit-economics](https://www.finops.org/framework/capabilities/unit-economics/)).  
> 2. **단위 혼합 합산 금지.** `ConsumedUnit`이 다른 수량(GB-시간·요청 수·GB 전송량 등)은 **이질적 물리량**이라 더할 수 없음.  
>    반드시 **서비스·미터·`ConsumedUnit`별로 분리 집계** 후 추이를 봄(실무 절차는 §6-① 참조).

## 6. 함정 (Pitfalls) — 반드시 확인

**① 단위 혼합 합산 금지 (가장 흔한 오류)**  
`ConsumedUnit`이 `Units`·`Hours`·`GB`·`GB/Month` 등으로 섞여 있어, 단위 고정 없이 `Sum(ConsumedQuantity)`를 내면  
"1 Hour + 1 GB = 2"처럼 **의미 없는 합**이 됨. **반드시 Unit 슬라이서로 단위 1개를 고정**한 뒤 추세를 봄(실측 분포 §7.2).

**② ConsumedQuantity vs PricingQuantity 혼동**  
사용량(수요) 추세는 **ConsumedQuantity** 로만 봄. `PricingQuantity`는 블록 프라이싱·구매 단위가 반영된 청구용 수량이라  
실제 사용량과 다름(§7.3 실측 약 57% 불일치). 단, `PricingQuantity` 컬럼은 **이 페이지 비주얼엔 없음**(개념 구분용).

**③ ChargeCategory = Usage 한정 원칙**  
사용량은 본질적으로 **Usage 행**에 존재함. `Purchase`(약정 선구매)·`Tax`·`Credit` 등은 사용량 추세에서 제외해야 함.  
실측(§7.1)에서도 대상 데이터는 사실상 전량 Usage임. 필요 시 `ChargeCategory` 필터로 Usage만 남김.

**④ 음수·조정·무료·시간내포 단위 해석 주의**  
- **음수 행**(credit·환불·조정)이 합계를 상쇄해 추세를 왜곡할 수 있음 → 분리 검토.  
- **무료 플랜**: 단가(Price)가 0이어도 사용량(ConsumedQuantity)은 존재할 수 있음 → 비용 0이라고 사용량도 0 아님.  
- **`GB/Month`·`Units/Month` 등 시간 내포 단위**: 단위 자체에 기간이 이미 포함됨. 월별 X축과 **이중 계산** 오해 주의  
  (해당 단위는 "월 환산 사용량"으로 해석).

## 7. 실측 예시 (EA-Cost-FOCUS_1.0 데이터셋)

> ⚠️ 아래 수치는 **예시 데이터셋(총 135,272행) 기준**이며, **절대 수치는 환경별 상이**함. 방법론 검증용 참고치임.

### 7.1 ChargeCategory 분포 → 사용량은 Usage에 집중

| ChargeCategory | 행 수 |
|---|---|
| Usage | 135,271 |
| Purchase | 1 |

→ 사용량 분석 대상은 사실상 전량 Usage 행. (§6-③ 원칙과 일치)

### 7.2 ConsumedUnit 분포 (Usage 행 기준) → 단위 혼합 실증

| ConsumedUnit | 행 수 |
|---|---|
| Units | 77,163 |
| Hours | 17,707 |
| GB | 14,976 |
| GB/Month | 12,738 |
| Units/Month | 9,659 |
| Units/Hour | 2,609 |
| Units/Day | 304 |
| GiB Hours | 58 |
| GB Hours | 57 |
| **합계** | **135,271** |

→ 9종 단위가 혼재. **단위 고정 없이 합산하면 무의미**(§6-①의 실증 근거).

### 7.3 ConsumedQuantity ≠ PricingQuantity → 약 57% 불일치

- 두 값이 다른 행: **77,012 / 135,271 ≈ 약 57%**.  
- 예: **Azure DNS** → `ConsumedQuantity = 0.000144 Units` vs `PricingQuantity = 0.000000000144 Units`  
  (배율 약 100만 배). 동일한 사용 사건을 서로 다른 수량으로 표현함(블록 프라이싱 영향).  

→ 결론: 사용량 추세는 **반드시 ConsumedQuantity** 로 봄. PricingQuantity로 보면 수요가 심하게 왜곡됨.

## 8. 체크리스트

- [ ] 리포트가 읽는 Export가 **FOCUS 1.0 정식**(`ConsumedQuantity`·`ConsumedUnit`)인지 확인함  
- [ ] **Unit 슬라이서(`ConsumedUnit`)로 단위를 1개 고정**한 뒤 사용량을 읽음 (혼합 합산 금지)  
- [ ] 사용량 추세는 **`ConsumedQuantity`** 로만 봄 (`PricingQuantity` 아님)  
- [ ] 오버레이 콤보차트에서 **비용 선(EffectiveCost)을 무시·제거**하고 막대(사용량)만 읽음  
- [ ] (편집모드) 비용 계열 없는 **사용량 전용 차트**(X=`ChargePeriodStart`, Y=`Sum(ConsumedQuantity)`,  
  범례=`ServiceName`·`x_SkuMeterCategory`)를 구성함  
- [ ] 범례를 서비스→미터로 바꿔 **추세 동인을 귀속**함  
- [ ] 급증월을 격리하고 **일/월 granularity 전환·미터 드릴다운**으로 이상 신호를 확인함  
- [ ] `ChargeCategory=Usage` 한정, **음수·무료·시간내포 단위** 행의 왜곡을 점검함  
- [ ] 결론에 "사용량 요인 vs 단가·할인 요인"을 **분리 서술**함
